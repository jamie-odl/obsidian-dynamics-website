DETECTION CLASSES

Eight Anomaly Types. Full Coverage.

Every detection is advisory, timestamped, and structured for analyst review — not a final determination.

Loiter

Aircraft remains in the same H3 cell beyond the configured dwell threshold. Flags circling, holding, or surveillance behaviour.

Ghost

Missing callsign with non-zero position and airborne altitude profile. Identifies aircraft transmitting position but no identification.

Squawk

Emergency transponder codes — 7500 (hijack), 7600 (radio failure), 7700 (general emergency). Immediate priority events.

Rapid Descent

Abnormal vertical speed exceeding safe descent parameters. Identifies potential emergencies or unusual operations.

ICAO Spoof

Aircraft transmitting a Mode-S address that doesn't match expected registration. Flags potential identity manipulation.

GPS Jamming

Position data inconsistencies suggesting GPS interference or spoofing. Cross-references multiple data points for validation.

Formation Flight

Multiple aircraft maintaining unusually close proximity and coordinated movement patterns outside of published formation routes.

Callsign Duplicate

Same callsign appearing simultaneously on multiple transponders. Indicates configuration errors or potential spoofing.

TRUST AND METHOD

How Detections Are Produced

01

Live Data Ingestion

SkyGrid continuously ingests global ADS-B and Mode-S data from distributed receiver networks, providing real-time position and metadata for thousands of aircraft simultaneously.

02

Watch Area Filtering

Analysts define geographic watch areas. Only aircraft inside active watch zones are processed for anomaly evaluation, keeping signal-to-noise ratio high and compute focused.

03

Rule Evaluation

Eight detection classes run in parallel against filtered traffic. Loiter dwell thresholds, ghost callsign checks, squawk code monitoring, rapid descent, GPS consistency, and more.

04

Event Delivery

Matched anomalies are classified, timestamped, and delivered via the dashboard, REST API, or webhook endpoints. Structured JSON payloads ready for downstream triage and integration.

METHODOLOGY

What Counts as a Detection

Every detection is advisory, not deterministic. Here's how we define, validate, and communicate anomaly events.

Baseline Logic

Each detection class uses rolling baselines (7/30/90/365-day windows) to establish normal behaviour per watch area. Deviations are scored against these baselines, not static thresholds.

Confidence Scoring

Every event includes a confidence score (0.0–1.0) derived from signal strength, data completeness, baseline deviation factor, and corroborating data points. Analysts filter by confidence.

False Positive Handling

Detections are advisory signals, not verdicts. Analysts review events in context. Common benign patterns (holding patterns near airports, training exercises) are documented in the knowledge base to support triage.

Known Limitations

ADS-B coverage varies by region. Aircraft without functioning transponders are invisible to the system. Jamming detection relies on position inconsistencies, not RF analysis. We document gaps, not hide them.

Data Source Provenance

All position data comes from distributed ground-based ADS-B receiver networks. Data is ingested via authenticated feeds, not scraped. Receiver network coverage maps are available in the knowledge base.

Latency Expectations

Typical end-to-end pipeline latency (receiver → ingestion → rule evaluation → delivery) is under 60 seconds. Webhook delivery adds <200ms. Dashboard refresh follows the same cycle.

ANALYST WORKFLOW

How Analysts Use SkyGrid

From alert to action in three steps.

01

Alert Arrives

A loiter detection fires for an unidentified aircraft circling over a watch area. The alert appears on the dashboard, arrives via webhook in Slack, and is logged in the event table with full metadata.

02

Triage in Context

The analyst opens the 3D map, sees the aircraft track, checks the confidence score (0.87), reviews the baseline deviation (13× normal dwell), and cross-references with known holding patterns. No match — this is anomalous.

03

Act or Escalate

The analyst marks the event as confirmed, adds a note, and forwards the structured JSON payload to the downstream SIEM or coordination channel. The event is now part of the auditable record.
